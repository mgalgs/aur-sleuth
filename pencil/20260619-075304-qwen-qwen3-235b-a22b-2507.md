## less-watch.sh

Status: UNSAFE

Summary: Arbitrary code execution via unvalidated lessc compilation

Details:

The script uses `inotifywait` in a loop to monitor file changes and automatically recompiles a CSS file using `lessc`. While automatic compilation is a legitimate development practice, the script executes `lessc` without sanitizing or validating input, and runs in a recursive (-r) mode on the entire directory tree. More critically, it operates in the current working directory derived from `$0` without proper escaping or quoting, which can lead to path injection if the path contains spaces or special characters.

Additionally, this script could be abused in a supply chain context if placed in an automated build or deployment system, as it allows for unattended execution of `lessc`—a Node.js-based compiler—on potentially untrusted or modified input files. An attacker who could modify any file in the monitored tree could trigger code execution during the compilation process if the `lessc` processor supports arbitrary JavaScript execution (which it does via LESS.js plugins or inlined JavaScript). This creates a vector for arbitrary code execution when file watches are enabled in development environments.

The line `while inotifywait -r $DIR; do lessc pencil.less pencil.css; done` is particularly risky because it does not quote `$DIR`, enabling shellword splitting, and re-runs compilation on *any* file change anywhere in the directory tree, not just the expected `.less` file. This broad monitoring increases attack surface unnecessarily.

## index.js

Status: UNSAFE

Summary: Misconfigured Electron security settings enabling RCE

Details:

The file exhibits multiple security issues that compromise the application's integrity and user safety. Most critically, it disables key Electron security features: `webSecurity: false`, `contextIsolation: false`, and `nodeIntegration: true` together allow injected or malicious scripts full access to the local filesystem and Node.js APIs, creating a high-risk attack surface. The app loads the main interface via `file://` protocol from `__dirname + "/app.xhtml"`, meaning any cross-site scripting (XSS) in that file grants immediate remote code execution.

Additionally, the application disables the sandbox with `app.commandLine.appendSwitch("no-sandbox");`, weakening process isolation. This, combined with relaxed file access policies (`allow-file-access-from-files`, `allow-file-access`), increases the potential damage from compromised renderers. While some settings may be intended for development (e.g., `--enable-dev`), they are enabled by default in ways that expose users to known vulnerabilities without clear justification or mitigation.

The code also attempts to read user configuration from `~/.pencil/config.json`, but does not sanitize input, potentially enabling path traversal or arbitrary code execution if configuration values are used unsafely elsewhere. While no explicit exfiltration or remote code download is visible here, these misconfigurations constitute genuinely dangerous behavior for a desktop application.
