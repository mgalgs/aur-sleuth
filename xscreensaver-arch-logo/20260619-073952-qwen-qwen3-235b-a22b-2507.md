## sign_update.rb

Status: UNSAFE

Summary: Command injection via unsanitized input in shell command

Details:

The script executes shell commands using backticks with unsanitized user input from `ARGV[0]` and `ARGV[1]`, which are passed directly into a shell command involving `openssl`. The line:

`puts `openssl dgst -sha1 -binary < "#{ARGV[0]}" | openssl dgst -dss1 -sign "#{ARGV[1]}" | openssl enc -base64``

constructs a shell pipeline using string interpolation. An attacker could supply a filename containing shell metacharacters (e.g., `;`, `$()`, `` ` ``) to inject arbitrary commands. For example, passing a filename like `"; rm -rf / ;"` could lead to remote code execution in the context of the user running the script.

While the script appears intended for signing archives, it does not validate or escape the input arguments before use in shell context, making it vulnerable to command injection. This constitutes a serious security flaw when processing untrusted input.

## update.sh

Status: UNSAFE

Summary: Downloads files over HTTP without verification

Details:

The script downloads `.po` translation files from an HTTP URL (`http://developer.gnome.org/projects/gtp/status/gnome-2.0-extras/po/xscreensaver.HEAD.`) using `wget` without any integrity checks or TLS encryption. The use of unencrypted HTTP allows for man-in-the-middle attacks, enabling an attacker to tamper with or inject malicious content into the downloaded files. While `.po` files are typically harmless, they could potentially be used in supply-chain attacks if later processed by a tool that interprets them in unsafe ways (e.g., through command injection if improperly handled during build).

Furthermore, the script silently replaces local `.po` files with remote content if differences are found, without user confirmation or verification. This behavior creates a vector for unauthorized code modification under the guise of translation updates. The remote URL points to what appears to be an outdated GNOME translation project infrastructure (`developer.gnome.org`), and the lack of modern security practices (HTTPS, checksums, signature verification) makes this script unsuitable for inclusion in a modern AUR package build process.

The core issue is the trust placed in an unsecured HTTP endpoint to provide code assets without validation. Even if the intent is harmless, this pattern undermines supply-chain security by introducing a weak link that could be exploited to introduce malicious payloads.

## installer.sh

Status: UNSAFE

Summary: Modifies and deletes user data across all accounts without consent

Details:

The script is designed for macOS and contains installer logic for XScreenSaver, but includes several dangerous operations that extend beyond standard packaging practices.

Notably, it performs system-wide file deletions using `rm -rf "$DSTVOLUME/Users/"*"/Library/Screen Savers/$f"`, which iterates over all user directories and unconditionally removes per-user screen saver installations. This is a privileged escalation risk and constitutes an unauthorized modification of user data outside the target installation path, potentially disrupting user preferences and configurations without consent.

Even more concerning, it later moves or renames preference files in per-user Containers directories using `mv "$OBH/org.jwz.xscreensaver."* "$NBH/"`, which manipulates user-specific application state across all users. While this might be intended to support migration, it runs with elevated privileges and makes irreversible changes to user data at an OS-protected path (`~/Library/Containers`), which is a security boundary violation. Such behavior opens the door to privilege escalation or data tampering if the installer is abused or run unintentionally.

Additionally, the script kills the `legacyScreenSaver` process with `kill -9`, which is an aggressive system modification that could destabilize the system or bypass expected lifecycle management. Combined, these actions go beyond application installation and perform system-wide intrusive operations without user confirmation or sandboxing, making this script unsafe.
