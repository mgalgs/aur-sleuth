## roxybrowser.desktop

Status: UNSAFE

Summary: Runs browser with sandboxing disabled by default

Details:

The `Exec` line in the desktop entry runs Roxy Browser with two critical Chromium security flags disabled: `--no-sandbox` and `--disable-setuid-sandbox`. These flags are essential for isolating the browser from the underlying system and preventing privilege escalation or arbitrary code execution in case of compromise. Disabling them by default in a desktop launcher significantly increases the attack surface and is considered a severe security anti-pattern.

While some specialized applications (e.g. antidetect browsers) may require relaxed sandboxing for functionality, distributing such a configuration in a system-wide `.desktop` file without explicit user consent or warning is dangerous. It automatically applies to all users and persists across launches, making it easy for users to unknowingly run the browser with elevated risks. This configuration effectively trusts all web content accessed through Roxy Browser with the user's full privileges.

The presence of only `--no-sandbox` and `--disable-setuid-sandbox` — with no other mitigating security policies (like seccomp, namespaces, or user namespaces) evident from this file — suggests a default-exposed posture. This is not merely a packaging oversight but an active weakening of security controls that should not be enabled by default in a distributed package.

## config.json

Status: UNSAFE

Summary: Hardcoded Google OAuth client secret exposed

Details:

The file `config.json` contains embedded Google API credentials, including an `apiKey`, `clientId`, and `clientSecret`. These secrets are hardcoded and exposed in a configuration file that is part of a public package distributed via the AUR. The presence of a valid Google `clientSecret` (e.g., "OTJgUOQcT7lO7GsGZq2G4IlT") is particularly concerning, as such secrets should never be included in client-side or public applications — they are intended for server-side use only. If compromised, they can be abused to impersonate the application in OAuth flows, potentially leading to token theft, account hijacking, or unauthorized access to user data.

This exposure constitutes a supply-chain risk because any user installing the package inherits these embedded secrets, which could be extracted and misused by malicious actors. Even if the credentials were previously valid for a legitimate application, their public exposure necessitates revocation. The inclusion of such secrets in open packaging is a well-known anti-pattern and a direct violation of security best practices. This is not a packaging standard but a critical oversight that enables abuse.
