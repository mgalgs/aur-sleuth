## binary-install.js

Status: UNSAFE

Summary: Downloads and runs binaries without integrity verification

Details:

The script `binary-install.js` downloads and executes code from GitHub releases without verifying cryptographic signatures or checksums, creating a supply-chain attack risk. It retrieves a binary archive from `https://github.com/sinelaw/fresh/releases/download/v${VERSION}/...` based on the local package.json version, which could be manipulated if an attacker compromises the package or its dependencies. The download is performed over HTTPS, but there is no integrity verification (e.g., checking SHA256 hashes) after downloading, making it vulnerable to man-in-the-middle or repository compromise attacks.

Furthermore, the script directly uses `execSync` to run system commands like `tar`, `unzip`, and PowerShell's `Expand-Archive` on the downloaded archive, which could lead to arbitrary code execution if the archive is tampered with. While the script targets a legitimate repository, the lack of any form of binary validation turns this into a potential vector for remote code execution during installation. This behavior is especially concerning in automated environments (e.g., CI/CD or npm install hooks), where malicious payloads could be silently executed.

## install.js

Status: UNSAFE

Summary: Auto-executes unverified binary installer

Details:

The file `install.js` imports and executes a function from a local module `./binary-install`. While importing local modules is common, the critical issue is that this script automatically runs `install()` without user interaction or clear opt-in, which is a known vector for supply chain attacks when combined with binary downloads.

More importantly, the referenced `binary-install` module is not a well-known, audited package from a public registry like npm, but rather a local file. This allows it to contain arbitrary logic, such as downloading and executing binaries from remote servers, modifying system files, or exfiltrating data. Without seeing the contents of `binary-install.js`, its behavior cannot be verified, but the pattern of auto-executing an installation routine that could pull binaries from unspecified sources is inherently risky.

This pattern violates the principle of least surprise and is commonly exploited in malicious packages to silently install payloads. Given that the AUR package might bundle or fetch such files, this automatic execution of a potentially hidden binary installer constitutes a serious security concern.
