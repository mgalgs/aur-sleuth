## add_stored_proc.py

Status: UNSAFE

Summary: Dynamic SQL and scheduled database events pose persistence and injection risks.

Details:

The script `add_stored_proc.py` creates a MySQL stored procedure and an event that automatically executes this procedure monthly to alter table partitions. While stored procedures and events can be part of normal database administration, this script poses a supply-chain risk due to its use of dynamic SQL execution within the database via prepared statements and scheduled execution.

The critical issue is the creation of a MySQL event (`eventCreatePartition`) that runs monthly and calls the stored procedure `CreateDataPartition`, which uses dynamic SQL formed via `CONCAT` and `PREPARE`. Although the dynamic SQL is currently used for legitimate partition management, the pattern of preparing and executing arbitrary SQL statements from constructed strings increases the risk of SQL injection if any inputs are ever tainted. Moreover, scheduled execution of dynamic SQL within the database runs outside the usual code review scope and can be used to persist malicious behavior.

While the script appears to serve a legitimate purpose (managing time-based table partitions), the combination of dynamic SQL construction, preparation, execution, and a scheduled database event that runs indefinitely without audit creates an elevated risk. These patterns are often exploited in supply-chain attacks to maintain persistence or escalate privileges. The file does not exfiltrate data or download remote payloads, but the use of advanced database features in a way that could hide malicious changes warrants marking this as UNSAFE due to the elevated risk profile.

## build.sh

Status: UNSAFE

Summary: Script auto-commits, pushes, and builds Docker images unsafely

Details:

The script performs several actions that fall outside the scope of a standard AUR package build process and exhibits behavior indicative of a supply-chain attack. It modifies and pushes to git repositories (including `git push` and `git push --tags`), which is inappropriate in a build script and suggests unauthorized changes to source control. It also runs `docker build` and `docker create` to build and extract a `.deb` package, potentially enabling the surreptitious inclusion of malicious binaries. Most critically, the script modifies the `changelog` and `files` via `ex` and `cat` in an automated, unreviewed way and commits and pushes those changes without user intervention.

Additionally, the script pulls from `upstream master`, fetches tags, and performs pushes — actions that can alter the source repository and introduce untrusted code. The use of `ex` in a heredoc to programmatically edit files increases opacity and obstructs review. While some actions like version detection and Docker usage could be part of a legitimate workflow, the combination of auto-committing, auto-pushing, unauthorized Docker image creation, and modification of packaging files without user confirmation indicates a dangerous deviation from secure packaging practices. This script appears designed to automate both packaging and deployment without sufficient oversight, creating a vector for unauthorized code injection.
