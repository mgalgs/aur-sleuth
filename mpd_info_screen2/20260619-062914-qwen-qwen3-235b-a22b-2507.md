Summary: Build script downloads code without safe verification

Details:

The CMakeLists.txt performs a risky network operation during the build process: it automatically downloads `raylib-6.0.tar.gz` from GitHub using `curl -L -o` if the file does not already exist locally. While downloading a dependency might seem normal, this is done unconditionally and without any integrity verification *before* extraction and building. Although a `sha256sum` check follows the download, it is part of the same command chain and could be bypassed if the build process is interrupted or manipulated.

More critically, the file uses `git describe --long --tags` in a custom target to generate version information. This runs `git` commands during the build against the current source directory. If this project is included as a submodule or built in a context where the `.git` directory is present and mutable, this poses a supply-chain risk. However, the bigger issue is the **download-and-execute pattern**: fetching a tarball during CMake configuration, applying local patches, and building it as a static library without using a verified or pinned external project mechanism.

While the SHA-256 check (2b3ee1e2120c7a...) provides some assurance, it occurs *after* the download in an unguarded command sequence. An attacker who compromises the build environment or intercepts the connection could replace the archive before the check runs. Furthermore, CMake is not designed to safely handle such dynamic downloads; it undermines reproducible builds and opens the door to time-of-check/time-of-use (TOCTOU) attacks.

This behavior represents a supply-chain vulnerability: the build script fetches and builds code from an external source without sufficiently guaranteeing the authenticity or integrity of that code at the point of download.
