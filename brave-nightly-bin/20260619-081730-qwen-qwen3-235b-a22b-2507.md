## postrm

Status: UNSAFE

Summary: Installs unverified PGP key and configures external APT repository

Details:

The `postrm` script performs post-removal actions for the `brave-nightly-bin` package but contains high-risk behaviors that extend beyond typical package cleanup. Most critically, it directly embeds and installs a large PGP public key (`PGP_KEY_DATA`) via `base64 -d` into `/usr/share/keyrings/brave-browser-nightly.gpg`. While PGP keys are often used for repository verification, distributing such a key *within a postinst/postrm script* instead of via a trusted package or secure channel is a red flag. The key is not checksum-verified before installation (despite `PGP_KEY_CHECKSUM` being present, no actual verification code uses it), allowing potential tampering.

Additionally, the script manipulates APT sources by creating or migrating to a `.sources` file pointing to `https://dl.google.com/linux/chrome-nightly/deb/`, a non-Brave, Google-controlled domain. This introduces a supply-chain risk: if Google's domain is compromised, an attacker could push malicious Brave updates. The script also modifies system behavior by creating symlinks in `/opt/brave.com/brave-nightly/` to system NSS and udev libraries — a technique sometimes used to bypass library version checks but which could be abused to intercept or manipulate sensitive cryptographic or hardware interactions.

Though some actions (like managing APT sources) are expected for a browser package, the combination of installing an unverified PGP key from within the script, pointing to a third-party domain (Google), and creating library symlinks constitutes suspicious and potentially dangerous behavior that exceeds standard packaging practices.

## postinst

Status: UNSAFE

Summary: Installs Google's key and repo as Brave's — supply-chain risk

Details:

The `postinst` script contains a critical security issue: it downloads and installs a public key and APT repository configuration for "Brave Browser (nightly)" from Google's Linux repository (dl.google.com), but the hardcoded PGP key and repository URL point to **Google's Chrome Linux repository**, not an official Brave source. The script attempts to install Chrome's signing key (owned by <linux-packages-keymaster@google.com>) under the name `brave-browser-nightly`, which is misleading and potentially dangerous.

The key itself is legitimate for Google Chrome packages, but its misuse here creates a supply-chain risk: an attacker could exploit confusion between Brave and Chrome sources, or the script could inadvertently lead users to trust Google's key for packages named as Brave. Worse, the script unconditionally installs this key and repository without verification that the sources actually serve Brave Nightly builds — which they do not. Actual Brave packages are hosted at `brave.com` (e.g., https://brave.com/static-assets/brave-packages-nightly/), not at `dl.google.com/linux/chrome-nightly/deb/`. 

Additionally, the script includes a base64-encoded PGP key (`PGP_KEY_DATA`) that is decoded and written to the system keyring via `echo "$PGP_KEY_DATA" | base64 -d`. While base64 decoding itself isn't inherently malicious, doing so with large, inlined blobs in postinstall scripts is a red flag because it obscures the true origin of cryptographic material. This pattern is often abused in malicious packages to hide backdoored keys or trojanized binaries.

Although the script exits early (`exit 0`) before executing some functions, the key installation and repository setup are performed unconditionally, making this a genuine supply-chain risk. The user ends up with a browser package named Brave but configured to use Google’s update infrastructure — a severe integrity violation.

```sh
echo "$PGP_KEY_DATA" | base64 -d >"$GPG_FILE.$$"
```

This command installs an unvetted, third-party cryptographic key into the system trust store, which can later be used to install arbitrary Google-signed packages under the guise of Brave updates.
