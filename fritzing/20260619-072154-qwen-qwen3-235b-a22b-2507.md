## release-linux.sh

Status: UNSAFE

Summary: Privileged Docker access with unsafe mounts

Details:

The script `release-linux.sh` executes a Docker container with the `--privileged` flag, which grants the container full access to the host system, including the ability to modify system devices and bypass all security boundaries. This is an extreme privilege escalation that is rarely justified in a release process and represents a severe supply-chain vulnerability.

Additionally, the script mounts the current working directory (`$(pwd)`) into the container, potentially exposing sensitive files (e.g. SSH keys, environment variables, configuration files) if the script is run from an unsafe directory. Combined with `--privileged`, this could allow the containerized process to modify host system state or extract data. The use of unvalidated environment variables like `$1` and `$2` as arguments to the container command increases the risk of command injection or misuse if not strictly controlled by a higher-level process. These practices violate least privilege and secure build hygiene.

## build-linux.sh

Status: UNSAFE

Summary: Uses unverified Docker image with full source access

Details:

The script uses `docker run` to execute `qmake` and `make` commands inside Docker containers with a volume mount of the current directory (`$(pwd)`) into the container at `/home/conan/fritzing`. While using Docker for building software is common, this script pulls and runs two containers (`fritzing/build:bionic`) from what appears to be a custom or private registry (no registry prefix is shown, so it defaults to Docker Hub), but there is no verification of the container's provenance or integrity.

More critically, the container is granted access to the entire current working directory, and the build commands are executed within that mounted context. If the `fritzing/build:bionic` image is compromised or replaced (e.g., via typo squatting or account compromise), it could modify or exfiltrate source code, steal credentials (e.g., SSH keys, GPG keys, environment variables if leaked), or corrupt build artifacts. The lack of image digest pinning (e.g., `@sha256:...`) means the script is not reproducible and could pull different image contents over time — a known supply chain risk.

Additionally, the script is not part of standard AUR packaging practices (which use `PKGBUILD`) and may be a maintainer helper, but even so, automatically executing Docker builds with writable host mounts without integrity checks on the image poses a significant security risk. Such behavior deviates from safe, auditable packaging norms and introduces an untrusted execution environment with full access to the source tree.
