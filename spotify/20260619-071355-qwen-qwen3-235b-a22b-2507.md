## postinst

Status: UNSAFE

Summary: Automatically adds trusted APT repo and keys without user consent

Details:

The script adds APT repository keys and modifies the system's APT sources list by writing a new source file to `/etc/apt/sources.list.d/spotify.list`. While adding a vendor's repository might seem normal for a proprietary application, this behavior goes beyond typical post-install responsibilities and introduces supply-chain risks. Specifically, the script silently adds `https://repository.spotify.com` as a package source without explicit user consent, which can lead to unmonitored installation of privileged packages with root access in the future.

Even more concerning, it performs this action non-destructively but automatically, bypassing standard system administrator control. The script checks for opt-out conditions (like presence of the file), but the default behavior—auto-enabling a persistent root-level package repository—creates a privilege escalation vector. If an attacker compromises `repository.spotify.com`, they could push malicious packages to all systems that ran this script. Additionally, copying `.gpg` keys from the package into `/etc/apt/trusted.gpg.d/` without validation further expands trust in an automated, user-unaware way.

This is not standard packaging behavior for AUR or even typical third-party binaries. Most AUR packages avoid modifying system package manager sources precisely to prevent such risks. Spotify's upstream may intend this for auto-updates, but implementing it in a postinst script distributed via AUR blurs the line between application and system-level trust, creating a legitimate supply-chain attack surface.
