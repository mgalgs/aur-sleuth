## direnv-setup.sh

Status: UNSAFE

Summary: Executes curl|bash and modifies user configs automatically

Details:

The script `direnv-setup.sh` performs multiple high-risk actions that constitute a supply-chain attack vector. It automatically installs `direnv` using package managers and, critically, falls back to downloading and executing an install script from `https://direnv.net/install.sh` via `curl | bash` without any integrity checks. This pattern is inherently dangerous, as it executes code from the internet directly in the user's environment, potentially allowing remote code execution by a malicious or compromised server.

Furthermore, the script modifies user shell configuration files (e.g., `.bashrc`, `.zshrc`) and user-level config files (e.g., `~/.config/direnv/direnv.toml`), injecting commands and settings automatically. It also runs `direnv allow`, which grants execution rights to local `.envrc` scripts — a feature known to be exploitable if the directory is shared or compromised. The script includes a comment referencing a PostHog Flox activation hook, suggesting it is part of an automated environment setup, but this does not justify unsandboxed, unreviewed modification of user configurations and execution of external scripts.

The unconditionally executed `curl -sfL https://direnv.net/install.sh | bash` is the most critical issue. This is a well-known anti-pattern in security due to the lack of verification, and when combined with automatic configuration injection and `direnv allow`, it significantly elevates the risk of unintended code execution and persistence.

## install-tool-versions.sh

Status: UNSAFE

Summary: Downloads and replaces uv binary without integrity checks

Details:

The script makes untrusted network requests to download and overwrite the `uv` binary from GitHub without verifying integrity. It downloads `https://github.com/astral-sh/uv/releases/download/${TARGET_UV}/uv-${UV_ARCH}.tar.gz` using `curl` and directly extracts and copies the binaries into the user's PATH without checksum verification or signature checking. This creates a supply chain vulnerability where an attacker compromising the GitHub repository or the connection via MITM could inject malicious code.

While downloading from GitHub (a known upstream) is expected, the script overwrites an existing binary (`/root/.local/bin/uv`) with elevated privileges and no validation. The use of `curl | tar` or direct binary replacement without integrity checks is a known anti-pattern in secure scripting. Additionally, it runs as root in a sandbox environment, amplifying the risk. Though tool version management is within scope, silently replacing core binaries without verification crosses into unsafe behavior.

The script also downloads Node.js binaries from `https://nodejs.org/dist/` — a legitimate source — and installs them after extracting to `/opt`, which is more acceptable due to the trusted domain and lack of silent execution. However, the `uv` binary download lacks even this level of trust calibration. Combined with environment manipulation via `CLAUDE_ENV_FILE`, this could persist malicious state.
