## docgen.py

Status: UNSAFE

Summary: Executes unvalidated file content using exec()

Details:

The file `docgen.py` uses `exec(open("dropbox").read(), env)` to dynamically execute the contents of a file named `dropbox` in a modified environment. This is a high-risk operation, as it runs untrusted code from a local file without validation. The `dropbox` file is not verified, and if an attacker can control or modify it (e.g., via a malicious build process or compromised upstream), arbitrary code can be executed during package building or documentation generation.

The use of `exec()` with file input is particularly dangerous because it can lead to arbitrary code execution, especially if the `dropbox` file is not part of a verified source tarball or its integrity is not assured via checksums. While `dropbox` might be a legitimate module in context, the pattern of reading and executing a file named `dropbox` (which could be confused with the Dropbox binary or client) from the local filesystem without clear safeguards or input validation represents a supply chain vulnerability. This could be exploited in a malicious fork or poisoned build environment to run stealthy payloads during package installation.

## dropbox.in

Status: UNSAFE

Summary: Downloads and runs unsigned proprietary binary without enforced signature verification

Details:

The script downloads and executes Dropbox binaries from `https://www.dropbox.com/download` without proper verification in all cases. While it attempts to verify the binary's signature using a hard-coded GPG public key, this verification is **optional** and fails open: if `python3-gpg` or `gpgme` is not installed (common on minimal systems), no signature check occurs and the download proceeds silently with only a warning printed (`GPG_WARNING`). This was explicitly noted in the code: "Note: python3-gpg [...] is not installed, we will not be able to verify binary signatures."

The `download()` function fetches a proprietary, closed-source binary directly over HTTPS and unpacks it into `~/.dropbox-dist` with execution permissions, then runs it. Since the signature verification can be trivially bypassed by the absence of a Python GPG module, an attacker with MITM capabilities or control over the build environment could inject malicious code during installation or update.

This behavior constitutes a supply chain risk: the script assumes TLS alone ensures authenticity and integrity, but does not enforce cryptographic signature validation — a critical flaw when installing untrusted, proprietary binaries.
