Summary: Unsafe direct tar extraction into filesystem

Details:

The script `install.sh` performs an unsafe installation by extracting files directly into the target directory (`$dst`) with elevated privileges, which can lead to system compromise. It uses `tar` to unpack arbitrary files without validating or filtering their paths, allowing for archive traversal or overwriting of system-critical files. For example, a malicious payload inside the archive could include paths like `../../../etc/passwd` or `../../../root/.ssh/authorized_keys`, enabling privilege escalation or remote code execution.

Further, the script does not verify the integrity or origin of the files being installed. It assumes the contents of `$src` (e.g. `./install` or `./dist/install`) are trustworthy, with no checksums, signature checks, or input sanitization. This creates a supply-chain risk: if an attacker gains control of the build or distribution process (e.g. via a compromised upstream), they can inject malicious files that get silently installed anywhere on the system when run as root.

A key red flag is the use of unguarded `tar` extraction followed by immediate unpacking into the root filesystem:
```
(cd $tmp; tar -cf - *) | tar --no-same-owner -C "$dst" -xf -
```
This pattern bypasses standard package manager safeguards (like file ownership and conflict detection) and is typical of standalone installers used in malware or backdooring scripts.
