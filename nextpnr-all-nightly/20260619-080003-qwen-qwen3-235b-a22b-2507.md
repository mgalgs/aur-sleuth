## picorv32.sh

Status: UNSAFE

Summary: Downloads unverified remote code and executes it via toolchain

Details:

The script downloads `picorv32.v` directly from GitHub using `wget` without any checksum verification or signature checking, making it vulnerable to a man-in-the-middle or upstream tampering attack. More critically, it executes `yosys` and `nextpnr-ice40` on the downloaded Verilog file, which could lead to code execution if the toolchain processes malicious constructs in the file. While the tools are intended for synthesis and place-and-route, complex HDL files can sometimes trigger unexpected behavior in EDA tools, especially if they contain crafted constructs. The lack of input validation, combined with execution of untrusted remote content through potentially vulnerable tooling, constitutes a supply-chain risk. The script also assumes the presence of `../nextpnr-ice40`, suggesting it may rely on externally built binaries, further reducing auditability and trust.

This script does not follow secure AUR packaging practices: it dynamically fetches code at runtime instead of using the `source` array and `md5sums`/`sha256sums` in a PKGBUILD, and performs network operations during execution—a red flag for reproducibility and security. Although `picorv32.v` comes from a legitimate upstream (cliffordwolf), the absence of integrity checks makes it easy for an attacker to substitute malicious content if the connection is compromised.

Example problematic line:
`wget https://raw.githubusercontent.com/cliffordwolf/picorv32/master/picorv32.v`
This downloads and uses a file without verification, then feeds it directly into synthesis tools that parse and execute logic—posing a potential risk if the file is tampered with.
