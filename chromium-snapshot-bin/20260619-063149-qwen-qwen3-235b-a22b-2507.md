The PKGBUILD contains a top-level command substitution using `curl` to fetch the latest Chromium build number from an external URL:

```
_build=$(curl -s "https://commondatastorage.googleapis.com/chromium-browser-snapshots/Linux_x64/LAST_CHANGE")
```

This command runs during the sourcing of the PKGBUILD, which occurs when `makepkg` is invoked—even with `--nobuild --nodeps --noprepare`. The result is that simply parsing the PKGBUILD triggers an untrusted network request to download data from Google's storage bucket. While the destination (`commondatastorage.googleapis.com`) is a legitimate Google Cloud Storage endpoint used by the official Chromium project, **the act of performing an uncontrolled network call during PKGBUILD parse time is itself a supply-chain risk**.

More critically, the value of `_build` is later used to construct source URLs for downloading binary blobs (the Chrome and Chromedriver ZIP files). Although those downloads happen later during the actual `makepkg` run (and would be validated by checksums), the initial `curl` call **runs unconditionally and its output influences which binaries are fetched**, without any integrity checks on the `_build` value itself. This creates a potential window for an attacker to poison the `LAST_CHANGE` file (via compromise of the storage bucket or MITM, though HTTPS mitigates the latter) and thereby force the download of a malicious build.

While the use of `curl` here serves a functional purpose (determining the latest snapshot), **executing network requests in the top-level scope of a PKGBUILD is dangerous and can be abused in supply-chain attacks**. This should be replaced with a static `pkgver()` function or a safer mechanism (e.g., using `source` for version metadata). As written, merely sourcing the PKGBUILD can trigger external, unguarded behavior that affects build inputs.
