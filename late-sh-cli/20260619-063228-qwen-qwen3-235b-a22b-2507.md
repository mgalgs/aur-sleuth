## setup_app.sh

Status: UNSAFE

Summary: Generates and uploads SSH private key to GitHub secrets

Details:

This script collects sensitive credentials—including an AI API key, S3 access keys, and a generated SSH host key—and uploads them directly to GitHub secrets using the `gh` CLI. While some of these operations might be intended for legitimate configuration setup, the script runs unconditionally and without sufficient safeguards, potentially enabling unauthorized exfiltration of credentials if an attacker gains control of the environment or repository.

The most critical issue is that the script generates an Ed25519 SSH host key locally using `ssh-keygen`, stores it in plaintext in a variable (`SSH_HOST_KEY`), and then uploads it as a GitHub secret via `gh secret set SSH_HOST_KEY`. This exposes the private key during script execution and may leave traces in shell logs or process lists. Additionally, the script prompts for and transmits an `AI_API_KEY` directly to GitHub secrets, which could be misused if the repository is compromised.

While the script does perform legitimate configuration tasks, the combination of generating and transmitting private keys and long-term credentials—without clear justification or mitigation—constitutes genuinely risky behavior. The use of `base64` to encode credentials for Docker config generation is standard but unnecessary since `gh auth token` and `gh api` are already used; however, this is not a primary concern. The core problem is the unnecessary generation and handling of private keys alongside tokens, increasing risk surface.

A safer approach would avoid generating keys in-line, rely on infrastructure-as-code practices (e.g., Terraform for key creation), or use GitHub Actions with OIDC for secure, token-free registry access.
