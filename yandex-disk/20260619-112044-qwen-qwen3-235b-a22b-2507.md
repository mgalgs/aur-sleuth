## postinst

Status: UNSAFE

Summary: Downloads GPG key over HTTP, insecure apt-key use

Details:

The script downloads a GPG key from a remote server (`YANDEX-DISK-KEY.GPG`) using either `wget` or `curl` and adds it to the system's APT keyring using `apt-key add`, which is a deprecated and insecure practice. While `apt-key` itself has known security implications (it adds keys system-wide without specifying a narrow origin), the critical issue here is that the script blindly trusts and imports a GPG key from an HTTP source (not HTTPS) — both `http://repo.yandex.ru` and `http://repo.mirror.yandex.net` use plain HTTP, making the download vulnerable to MITM attacks. An attacker could intercept or modify the key, then sign malicious packages that APT would accept as legitimate. Furthermore, the script adds a repository entry to `sources.list.d/` pointing to this unsecured source, enabling persistent installation of untrusted or tampered packages. Although the domains are related to Yandex, the lack of transport security (HTTPS) and absence of key fingerprint validation make this an unsafe supply-chain vector.

Additionally, the use of `mktemp` to create a temporary file for the key is correct, but the overall trust model is flawed: downloading a cryptographic key over an unencrypted, unverified channel defeats its purpose. Modern systems should use `curl | gpg --dearmor` into `/etc/apt/trusted.gpg.d/` with verification or use signed repositories over HTTPS. This script bypasses those protections, increasing the risk of package tampering.
