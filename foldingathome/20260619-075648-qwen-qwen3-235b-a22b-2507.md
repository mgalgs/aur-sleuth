## update_boost.sh

Status: UNSAFE

Summary: Downloads and builds unverified code from remote source

Details:

The script downloads a 7z archive from `https://boostorg.jfrog.io/artifactory/main/release/1.83.0/source/boost_1_83_0.7z` using `wget` without any checksum verification. While using `wget` and extracting files is normal, the lack of integrity checking on the downloaded file is a hygiene issue, but not the primary concern.

The critical red flag is the execution of `./bootstrap.sh` and `./b2 tools/bcp` inside the extracted Boost directory — these are build instructions executed on code that is both externally fetched and not pinned to a verified commit or hash. The script pulls Boost directly from a remote server during execution without verifying the authenticity or integrity of the source. Since the `URL` is constructed dynamically and fetched unconditionally from a mutable release endpoint, an attacker who compromises the host or performs a MITM attack could substitute a malicious payload. The subsequent build and use of `bcp` (Boost Copy tool) from this unverified source introduces a supply-chain risk where arbitrary code could be executed during the build or embedded into the extracted files.

Although the script appears to support a legitimate use case (extracting Boost subsets), it does so in a way that introduces high-risk behavior: downloading, building, and executing code from an external source without verification. This constitutes a potential supply-chain attack vector.

## Makefile

Status: UNSAFE

Summary: Deploys to suspicious openscam.org domain with root access

Details:

The Makefile contains a `publish` target that uses `rsync` to synchronize files to a remote server at `root@openscam.org:/var/www/cbang.org/http/`. The domain "openscam.org" is highly suspicious, with "scam" in the name, strongly indicating a malicious or untrustworthy destination. Deploying code to such a domain, especially using root-level SSH access, presents a significant security risk and could facilitate the distribution of malicious content.

While the rest of the Makefile performs typical build tasks (compiling Jade templates), the publish target enables unauthorized deployment to a questionable remote host, which could be exploited for malicious purposes if executed.

This file does not appear to be a standard AUR package build script but rather a maintainer deployment tool. Its presence in a package build context raises red flags, as automated deployment to an external, potentially malicious server is not part of normal packaging workflows and could enable supply chain attacks.
