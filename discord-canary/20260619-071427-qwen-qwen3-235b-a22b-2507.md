Specifically, it calls an `updater_bootstrap` binary that is searched for in `/usr/share/`, `/opt/`, or the script's own directory, and then uses it to download the application from `https://updates.discord.com/` into the user's `$XDG_CONFIG_HOME` directory. The key issue is that this bootstrap process runs with no integrity verification (e.g., no checksum or signature check), downloads from the internet at runtime, and the downloaded content is immediately executed.

Furthermore, the script writes executable binaries into the user's config directory (`~/.config/discordcanary`) and runs them directly. This pattern creates a supply chain risk: if `updates.discord.com` is compromised or if DNS is hijacked, an attacker could deliver a malicious payload. While `updates.discord.com` is the official Discord update domain, the dynamic execution of downloaded code without cryptographic verification constitutes a security risk. Worse, the bootstrap binary itself is not verified and may be self-signed or lack transparency.

This behavior matches risky patterns seen in past supply-chain attacks: unverified runtime downloads of executable code, lack of pinned checksums, and direct execution of user-writable paths. Even if upstream is legitimate, the design enables future malicious updates to go undetected. Thus, the script is marked **UNSAFE** due to the potential for remote code execution if the update server is compromised.
