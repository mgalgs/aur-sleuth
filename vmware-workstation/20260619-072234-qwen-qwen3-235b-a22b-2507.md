## unlocker.py

Status: UNSAFE

Summary: Modifies VMware binaries with obfuscated data and unsafe patching

Details:

The script modifies core VMware binaries (vmware-vmx, vmwarebase, etc.) by patching memory pointers and writing obfuscated data into the files. Specifically, it locates OSK0 and OSK1 keys in the binary and writes rotated (ROT13) strings:
<pre>
bheuneqjbexolgurfrjbeqfthneqrqcy  --> "osx decrypt these words guarded by"
rnfrqbagfgrny(p)NccyrPbzchgreVap  --> "please dont steal(c)Apple Computer Inc"
</pre>
While the decrypted messages appear harmless, the act of modifying signed binaries in this way constitutes a supply-chain tampering risk. The script alters internal structures (vSMC key tables and RELA entries) without verification or safety checks, and enables functionality (macOS on non-Apple hardware) that VMware does not support. These actions could introduce instability or security vulnerabilities, and the patching mechanism itself — especially rewriting ELF .rela.dyn sections — could be abused to inject malicious code. The lack of integrity checks, combined with binary rewriting and obfuscation (ROT13 as code), meets the criteria for a potentially malicious payload.

## services.sh

Status: UNSAFE

Summary: Executes unvalidated script in system directory

Details:

The script `services.sh` executes a bash script located at `/etc/vmware-tools/services.bash` using `/bin/bash` with unvalidated arguments (`"$@"`). The primary issue is that it calls `bash` to execute a script in a system directory that may be writable or modifiable by other components or attackers, creating a risk of arbitrary code execution if the target script is tampered with.

Furthermore, the use of `/bin/bash` to invoke a script that may not have executable permissions or may be unexpectedly modified bypasses certain security assumptions. This pattern can be abused if an attacker gains control of `/etc/vmware-tools/`, a world-writable or weakly secured directory in some configurations. VMware Tools has historically used such mechanisms to manage guest integration, but invoking a central script via `bash` directly — especially without integrity checks — increases attack surface.

While this may reflect normal VMware Tools behavior, the pattern constitutes a supply-chain risk: a malicious actor could replace or modify `services.bash` to escalate privileges or persist malware. The script does not validate the origin or integrity of the target file, making it a potential vector for abuse.
