## modprobed-db_timer.in

Status: UNSAFE

Summary: Infinite loop for scheduling creates opaque, uncontrolled execution

Details:

The script runs an infinite loop that executes `/usr/bin/modprobed-db storesilent` every 6 hours using a backgrounded shell loop with `sleep 6h`. While periodic execution of a legitimate command may seem benign, the implementation is problematic: it uses `command="/bin/sh"` with `command_args` containing an unbounded `while true` loop, which is not a standard or safe way to implement periodic tasks in OpenRC. This pattern bypasses normal job control and logging mechanisms, making it difficult to audit or terminate cleanly.

More critically, the approach mimics a cron daemon but does so outside the supervision of standard scheduling systems (cron or systemd timers), increasing the risk of process proliferation or silent failure. While `/usr/bin/modprobed-db` is a legitimate tool for tracking kernel module usage, the method of execution here — a manually forked, long-lived shell loop — is atypical and could be exploited if the `modprobed-db` binary were ever compromised. Combined with `command_background=true` and direct `pidfile` management, this creates an opaque, persistent execution environment that deviates significantly from secure service design.

The comment mentioning "crontab equivalent" confirms that this is meant to perform scheduled tasks, but OpenRC is not designed for such fine-grained periodic scheduling. The correct approach would be to use `cronie`, `dcron`, or `fcron` with a standard crontab entry. This file's method suggests poor operational hygiene and introduces unnecessary risk by forking persistent background processes outside proper supervision.
