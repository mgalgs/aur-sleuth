## close-docker.sh

Status: UNSAFE

Summary: Dangerous unquoted variable and forceful container removal

Details:

The script contains a dangerous command that uses `docker rm --force` on containers matched by name without sufficient validation. The line `containers=`docker ps | grep "gnome-shell-system-monitor-applet" | awk '{ print $1 }'`` captures container IDs based on a partial name match, which could potentially match unintended containers if the name substring appears in other container names. More critically, the script runs `docker rm --force $containers` without quoting the variable (`"$containers"`), which can lead to argument injection or word splitting if the output contains unexpected characters or multiple entries. While the use of Docker commands may be intended for development or cleanup purposes, the lack of input sanitization, error handling, and reliance on fragile text parsing makes this script risky, especially if executed in an environment where container names can be influenced by untrusted input. This pattern could be exploited to disrupt services or remove critical containers.

## upload.py

Status: UNSAFE

Summary: Handles plaintext credentials for website login

Details:

The provided `upload.py` script automates interactions with the GNOME Extensions website to log in and upload an extension. It retrieves credentials (username and password) from environment variables (`USERNAME`, `PASSWORD`) and uses them to authenticate via Selenium. The use of Selenium with headless Chrome to perform login actions using credentials passed in plaintext poses a significant security risk, especially if this script is executed in an untrusted context or logs environment variables.

More critically, the script sends user credentials to a third-party website (`https://extensions.gnome.org`) without any indication of encryption, multi-factor authentication, or secure handling guarantees. While interacting with legitimate services isn't inherently malicious, this script could be abused in a supply-chain attack to capture and transmit credentials if the target site were spoofed or if the script were modified to log inputs. Furthermore, automated credential-based login to external services is a dangerous practice and contradicts modern authentication standards (e.g., token-based or OAuth flows).

Although the script appears to serve a legitimate development purpose (uploading a GNOME extension), its inclusion in an AUR package — especially one meant for end-user installation — raises red flags. AUR packages should not include scripts that require or handle user credentials, particularly when they automate logins using environment variables. This crosses from packaging functionality into sensitive credential handling without appropriate safeguards, making it unsafe.

```python
username = os.getenv('USERNAME')
password = os.getenv('PASSWORD')
```

These lines extract credentials in plaintext, which could be logged, intercepted, or exfiltrated if the environment is compromised.

## run_and_kill.py

Status: UNSAFE

Summary: Executes unverified local scripts using os.system

Details:

The script executes `./run-docker.sh` and monitors its output, but includes a dangerous fallback: if the process becomes "stale" (i.e., timeout exceeded after detecting the STARTED_TAG), it runs `os.system('./close-docker.sh')`. While that may seem benign, the use of `os.system()` is unsafe and unnecessary—proper process control could use `subprocess` with explicit arguments. More critically, the script runs shell scripts (`run-docker.sh` and `close-docker.sh`) from the local directory without verifying their contents or origin. This creates a supply-chain risk: if an attacker compromises or plants one of these scripts (e.g., via a malicious PR or compromised build environment), the script will silently execute arbitrary commands with the same privileges as the user.

Furthermore, the script changes to the script's directory using `os.chdir(os.path.dirname(os.path.abspath(__file__)))`, increasing the attack surface by ensuring relative script paths resolve to the package's source tree. This behavior, combined with executing local shell scripts without integrity checks, constitutes a supply-chain vulnerability. While no explicit network exfiltration or base64-encoded payloads are present, the pattern of unsandboxed, unverified script execution using `os.system` on local files makes this risky in a distributed package context.

There is no checksum, source verification, or containment of these helper scripts in the provided code, and their behavior is not audited. In the context of an AUR package, where source integrity is paramount, this pattern could be abused to execute malicious payloads during installation or runtime if the scripts are tampered with.

## run-docker.sh

Status: UNSAFE

Summary: Excessive Docker privileges suggest container breakout risk

Details:

The script requests dangerous Docker capabilities: --cap-add SYS_BOOT and --cap-add SYS_ADMIN. These grant the container the ability to reboot the system, load kernel modules, and perform other privileged host-level operations. Such privileges are excessive for running a typical GNOME shell extension and can be abused to escalate privileges or interfere with system operation.

Additionally, the container is mapped to /sys/fs/cgroup, which exposes the host's cgroup hierarchy. Combined with SYS_ADMIN, this could allow the container to manipulate system services and escape isolation. No part of the application's described purpose justifies these capabilities, making their inclusion highly suspicious and a potential vector for container breakout or system manipulation.

The use of --rm suggests ephemeral runs, but the combination of privileged capabilities, host volume mounting (./testing), and network port exposure (5901, 6901) increases the risk surface, especially if the shared directory or VNC ports are used to exfiltrate data or gain interactive access.
