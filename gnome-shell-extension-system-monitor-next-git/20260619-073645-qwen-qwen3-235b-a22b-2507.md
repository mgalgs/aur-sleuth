## run-docker.sh

Status: UNSAFE

Summary: Elevated Docker privileges and unverified image source

Details:

The script uses `docker run` with `--cap-add SYS_ADMIN` and `--cap-add SYS_BOOT`, which grants elevated privileges that can allow container breakout or host system manipulation. It also binds `/sys/fs/cgroup` into the container, enabling access to cgroup control, which is dangerous when combined with `SYS_ADMIN`. These configurations are typically unnecessary for standard testing and significantly increase attack surface.

Additionally, the container is given access to the host's cgroup filesystem and mounts several tmpfs volumes, suggesting it may attempt to simulate a full system boot. While not inherently malicious in development contexts, these capabilities — especially when combined — are commonly exploited in container escape attacks. The lack of clear justification in comments and the script's integration into a package build/test flow raises supply-chain risk.

The image being run — `gnome-shell-system-monitor-applet` — is not pulled from a verified registry, and no image digest is pinned. This means any execution of the script pulls whatever version is locally available or from an unspecified source, allowing potential execution of unreviewed or tampered code.

## run_and_kill.py

Status: UNSAFE

Summary: Executes unverified local scripts with os.system

Details:

The script executes `./run-docker.sh` via `subprocess.Popen` and monitors its output. When a timeout condition is met (after 120 seconds of inactivity since the `STARTED_TAG` is seen), it runs `./close-docker.sh` using `os.system()`. While the use of `os.system()` is not inherently malicious, the combination of running shell scripts based on process output and timing conditions raises concerns about context and intent.

More critically, the script is designed to execute and manage Docker-related operations without any clear validation or constraints on what these shell scripts do. The fact that `run-docker.sh` and `close-docker.sh` are executed from the local directory—implying they could be modified or replaced—creates a risk of arbitrary command execution. An attacker could tamper with these scripts if the directory is writable, leading to code injection or privilege escalation.

Although this might be a legitimate monitoring script, it lacks safeguards such as script integrity checks or absolute path resolution, and runs potentially sensitive operations in a way that could be abused in a supply-chain attack. The script’s behavior—running, monitoring, and killing processes based on output content—combined with use of `os.system()` on local scripts, deviates from safe automation practices and introduces exploitable risk vectors.
