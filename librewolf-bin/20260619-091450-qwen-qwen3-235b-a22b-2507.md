## pack_vs.py

Status: UNSAFE

Summary: Downloads unverified binaries based on external YAML manifest

Details:

The script `pack_vs.py` downloads and processes Visual Studio packages based on a YAML manifest provided as input, then repackages them into a compressed tarball with a VFS overlay for Clang. While this functionality might appear legitimate in a development tooling context, the script calls `downloadPackages` and `extractPackages` from an imported module `vsdownload`, which is not part of the standard library or any widely known public package. No source or audit trail for `vsdownload` is provided in the file, and there is no verification of the origin, integrity, or authenticity of the downloaded Visual Studio artifacts.

More critically, the script allows arbitrary YAML input via the `manifest` argument, which is parsed using `yaml.safe_load`. However, `safe_load` is not invulnerable—especially if custom constructors or unsafe types have been previously loaded in the environment or if the `yaml` module has been monkey-patched. Though `safe_load` is used correctly here, combining it with download and extraction of unverified binary content into temporary directories, followed by repackaging and world-writable file creation, creates a risk of supply-chain compromise. The final output includes executable files whose sources are not pinned, verified, or auditable within this script.

This script poses a supply-chain risk because it operates on unverified network inputs and external YAML configuration to build a binary artifact without checksums, signatures, or source validation. It could be abused to inject malicious payloads into the output archive if the manifest or download sources are tampered with or attacker-controlled. The lack of origin checks, combined with automatic file extraction and re-archiving under manipulated paths, makes this script a potential vector for malicious repackaging.

## always-fetch-latest-toolchain-artifact.patch

Status: UNSAFE

Summary: Forcibly fetches unsigned build artifacts from remote CI

Details:

The patch modifies Mozilla's build system to forcibly fetch the latest toolchain artifact from the `mozilla-central` repository over HTTPS during the build process, using an unverified response from a hardcoded URL. Specifically, it introduces `requests.get()` to download `label-to-taskid.json` from `https://firefox-ci-tc.services.mozilla.com`, parses the JSON, and uses the result to override the expected task ID. This bypasses integrity checks and allows the build process to use toolchain components from the latest `mozilla-central` decision task, regardless of the current source state.

While the domain appears legitimate (Mozilla's own CI infrastructure), the behavior represents a supply-chain risk: it silently replaces pinned or expected build dependencies with externally fetched, non-reproducible artifacts. The comment even acknowledges the use of a less secure approach ("Using index.findArtifactFromTask for this would be nicer [...] but that doesn't follow redirects, so here we are"), indicating a known workaround that sacrifices security for convenience. This creates a window for tampering if the Mozilla CI endpoint were ever compromised or spoofed, and undermines the principle of deterministic, auditable builds.

The use of `requests` to fetch executable or build-affecting data without checksum verification or user control is dangerous in a build script. Although the target domain is owned by Mozilla, the change injects an uncontrolled network call into the build process that can alter build inputs without user awareness or consent—behavior typical of dependency confusion or supply-chain hijacking attacks.
