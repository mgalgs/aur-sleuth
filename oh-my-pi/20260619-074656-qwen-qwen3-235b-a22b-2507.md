## reload-runner.sh

Status: UNSAFE

Summary: Downloads and runs unsigned binaries over HTTP

Details:

The script establishes an SSH connection to a remote host specified by `CI_HOST` and executes a large block of inline bash code with elevated privileges (running as root, given paths like `/root/omp-kata-runner-image`). While much of the script is dedicated to building and deploying a container image, it downloads and executes binaries from third-party GitHub releases without verifying their integrity beyond the TLS channel.

Specifically, the script uses `curl -fsSL` to download BuildKit and nerdctl binaries directly from GitHub releases:
```
curl -fsSL "https://github.com/moby/buildkit/releases/download/v${BUILDKIT_VERSION}/buildkit-v${BUILDKIT_VERSION}.linux-amd64.tar.gz" -o "$archive"
curl -fsSL "https://github.com/containerd/nerdctl/releases/download/v${NERDCTL_VERSION}/nerdctl-${NERDCTL_VERSION}-linux-amd64.tar.gz" -o "$archive"
```
These binaries are then extracted and executed in the remote environment with no checksum verification. This constitutes a supply chain risk: if an attacker compromises the GitHub account or performs a URL hijacking attack (e.g., via typo-squatting or DNS compromise), they could inject malicious code into the build toolchain, leading to persistent access or backdooring of container images.

Although the script performs reasonable operational tasks (building images, updating Helm charts), the lack of cryptographic verification when downloading and executing binaries from external sources elevates this to a genuine security risk that could be exploited in a supply chain attack.

## tune-kata-runtime.sh

Status: UNSAFE

Summary: Remote config tampering with unvalidated input

Details:

The script performs potentially dangerous actions by modifying system configuration files on a remote host via SSH and executes arbitrary commands without proper input validation. It uses `ssh` to run a heredoc script on the remote host that:
1. Modifies the Kata Containers configuration file (`configuration-qemu.toml`) with user-controlled values for `default_vcpus`, `default_memory`, and `virtio_fs_extra_args`.
2. Uses Python to perform regex-based replacements in the config file without sanitizing inputs, enabling potential injection if those values are crafted maliciously.
3. Creates a Kubernetes pod with a dynamically fetched image from an autoscaling runner set, then executes commands inside it.

The most critical issue is the blind use of user-provided values in a system-level configuration file on a remote host. While the script appears to be for internal CI use, it lacks safeguards against malicious or malformed environment variables (e.g., `BOOT_VCPUS`, `VIRTIOFSD_THREAD_POOL`) that could lead to arbitrary code execution during the Python regex replacement or `kubectl` calls. Additionally, using `kubectl exec` to run `bash -lc 'bun --version; rustc --version | head -1'` inside a dynamically created pod based on a fetched image introduces supply-chain risks if the image is compromised. The combination of remote system modification, lack of input validation, and dynamic command execution in a privileged context (Kubernetes cluster) makes this script unsafe.
