## setup.sh

Status: UNSAFE

Summary: Unresolved template enables arbitrary JAR execution

Details:

The script uses `@PKG_NAME@-*.jar` in a command invocation, which suggests the use of template placeholders (commonly replaced during packaging via sed or similar tools). However, if this placeholder is not properly substituted before execution, the literal `@PKG_NAME@-*.jar` will be expanded by the shell as a glob pattern, potentially matching unintended JAR files in the current directory. This could lead to executing an unauthorized or malicious JAR file if an attacker places a file matching the pattern (e.g., `@PKG_NAME@-evil.jar`) in the same directory.

While the use of `export JAVA_HOME=./jre` and calling a bundled Java runtime is normal for self-contained applications, executing a JAR with an unresolved template variable introduces a risk of arbitrary code execution. The pattern `@PKG_NAME@` is typically used in AUR PKGBUILD templates and should be replaced during installation. If left unexpanded, it indicates a packaging error that could be exploited. This constitutes a supply chain vulnerability where incomplete template processing results in unsafe dynamic execution.

Example problematic line:
```
${JAVA_HOME}/bin/java -jar @PKG_NAME@-*.jar
```
The unresolved `@PKG_NAME@` combined with glob expansion creates an execution risk.

## build-jitsi-rpm.sh

Status: UNSAFE

Summary: Unpinned git pull and secret leakage in build

Details:

The script performs a `git pull --rebase` inside the `jitsi` directory after cloning it once, which introduces a supply chain risk by fetching and applying remote changes without integrity verification. This operation occurs in the build process and is not pinned to a specific commit, tag, or branch, allowing arbitrary unverified code to be included in the build.

More critically, the script substitutes values into the RPM spec file using `sed`, including credentials passed as command-line arguments: `-Dgoogle.api.client.id=$googleApiId -Dgoogle.api.client.secret=$googleApiSecret`. These credentials are embedded directly into the build command and may end up exposed in build logs, process lists, or the final package, creating a risk of secret leakage. Passing secrets via command line is insecure and violates secure build practices.

Additionally, the script downloads a source zip from `https://download.jitsi.org/jitsi/nightly/src/$zipFileName`, but the generated zip is not verified against this URL, and the actual source used is from the locally modified git repository. This mismatch could lead to inconsistency or tampering. The combination of untrusted code updates, secret exposure, and lack of reproducibility constitutes a security risk.
