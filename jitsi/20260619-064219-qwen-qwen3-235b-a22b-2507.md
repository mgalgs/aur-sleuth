## setup.sh

Status: UNSAFE

Summary: Wildcard JAR execution risks code injection

Details:

The script uses `@PKG_NAME@-*.jar` in a command substitution without proper quoting or validation, which can lead to command injection or unintended execution if multiple JAR files are present or if filenames are crafted maliciously. The use of a glob (`*`) in `${JAVA_HOME}/bin/java -jar @PKG_NAME@-*.jar` is dangerous because it assumes exactly one matching JAR file, but if an attacker can plant a malicious JAR file in the same directory (e.g., via symlink or write access), the wrong JAR could be executed.

Worse, `@PKG_NAME@` is a placeholder typically replaced during packaging. If the build process fails to substitute it correctly, the literal string `@PKG_NAME@` may remain, leading to a wildcard that expands to any JAR file (e.g., `./jre/bin/java -jar jitsi-*.jar` becomes `./jre/bin/java -jar *.jar` if PKG_NAME is undefined). This behavior weakens integrity controls and could allow execution of arbitrary code during setup. While the script appears minimal and may be part of a larger package, it lacks safeguards expected in a secure context.

## build-jitsi-rpm.sh

Status: UNSAFE

Summary: Unsafe: Unpinned git pull and secret leakage in build

Details:

The script performs a `git pull --rebase` inside a previously cloned repository (`jitsi`) without verifying the integrity or origin of the changes. While the initial clone is from a legitimate upstream (`https://github.com/jitsi/jitsi.git`), the `git pull --rebase` in the build process fetches and applies remote changes at build time that are not audited or pinned to a specific commit. This creates a supply-chain risk: an attacker who gains write access to the upstream repository could inject malicious code that would be silently included in the built RPM without detection.

The combination of unchecked code updates and secret handling makes this script unsafe for production or automated builds.
