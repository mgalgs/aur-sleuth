Summary: Unsafe git pull after checkout allows code injection

Details:

The script performs a `git pull` inside the `squashfuse` directory after checking out a specific commit. This operation can lead to arbitrary code execution if the remote repository is compromised or if a man-in-the-middle attack occurs, because `git pull` combines both `fetch` and `merge`. The use of `git pull` after `git checkout master` effectively allows unverified code updates at build time, bypassing the integrity of the pinned commit reference (`SQUASHFUSE_REF`). This violates supply-chain security principles by introducing mutable, unverified content during build.

Although the script attempts to pin a specific commit hash (`775b4cc72ab47641637897f11ce0da15d5c1f115`), the unconditional `git pull` in the `if` block undermines this assurance. In supply-chain attack scenarios, such patterns have been exploited to inject malicious payloads during the build phase. While the script operates within the package's own upstream GitHub repository, the `git pull` after `checkout master` enables arbitrary code execution from a potentially tampered upstream branch. This constitutes a genuine supply-chain risk and deviates from secure packaging practices that require fully reproducible builds.

A safe alternative would use `git fetch` followed by `git reset --hard`, or simply avoid network operations entirely after cloning. The current implementation combines network access (`git pull`) with direct execution of fetched code (via checkout), creating an unacceptable risk.
