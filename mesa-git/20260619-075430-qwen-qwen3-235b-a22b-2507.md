## kdl.sh

Status: UNSAFE

Summary: Hardcoded unvalidated external code execution

Details:

The script unconditionally sources a file from a hardcoded global path `/ci-kdl/bin/activate` and then executes `ci-kdl` from the same location. This introduces a critical security risk because:

- It trusts and executes code from a fixed, system-wide path (`/ci-kdl`) without validating ownership, integrity, or authenticity.
- An attacker with write access to `/ci-kdl` could replace the virtual environment or the `ci-kdl` binary to execute arbitrary code with the same privileges as the user running this script.
- The use of `source /ci-kdl/bin/activate` can modify the current shell environment maliciously (e.g., by setting trap handlers, overriding built-in commands, or exporting functions).

This behavior is particularly dangerous in a build or CI context where privilege escalation or persistence could be achieved. There is no verification that `/ci-kdl` is the intended environment, making this susceptible to path hijacking or trojan horse attacks.

## container_job_trampoline.sh

Status: UNSAFE

Summary: Arbitrary code execution via unvalidated env var

Details:

The script uses `eval "$S3_JWT_FILE_SCRIPT"` to execute arbitrary code from an environment variable, which is a severe security risk. The variable `S3_JWT_FILE_SCRIPT` is only checked for existence via `: "$S3_JWT_FILE_SCRIPT"` but not sanitized or validated, allowing execution of untrusted, attacker-controlled code. This pattern is commonly exploited in supply chain attacks to inject malicious logic.

Additionally, the script unsets `S3_JWT_FILE_SCRIPT` after evaluation, obscuring audit trails, and sets a trap that deletes `${S3_JWT_FILE}` on exit — yet `S3_JWT_FILE` is never defined in the script, suggesting reliance on external, possibly hidden, state. The combination of `eval` on an environment variable and unclear file handling indicates high risk.

Example of dangerous line:
```
eval "$S3_JWT_FILE_SCRIPT"
```
This allows arbitrary command injection if the environment is compromised, such as executing remote payloads or exfiltrating data.

## weston.sh

Status: UNSAFE

Summary: Launches display server with potential session hijacking risk

Details:

The script initiates a Weston compositor instance with a hardcoded display socket and log path, but the critical issue is the `while [ ! -S /tmp/.X11-unix/X0 ]; do sleep 1; done` loop, which waits for an X11 Unix socket to appear. This indicates an attempt to bridge or interact with an X11 server, potentially enabling access to active graphical sessions.

More concerning is that the script runs Weston in the background (`&`) without proper sandboxing, drops privileges implicitly, and waits indefinitely on a socket in `/tmp/.X11-unix`, a world-writable directory. This could facilitate session hijacking, interception of input events, or impersonation attacks if combined with other tools. While Weston itself is legitimate, this script could be abused in a broader attack to capture user input or escalate within the display server context.

The script does not exfiltrate data directly, but its behavior—launching a display server with broad access and waiting for X11 sockets—goes beyond typical packaging or testing scripts and suggests potential misuse in a supply chain attack to compromise user sessions.

## init-stage1.sh

Status: UNSAFE

Summary: DNS manipulation via hardcoded IP

Details:

The script contains a conditional network configuration that sets a custom DNS server (192.19.189.10) for a specific environment (`FARM=vmware`). While setting DNS temporarily might be part of internal automation, the hard-coded IP address in a global configuration file (`/etc/resolv.conf`) is concerning because it lacks authentication or access controls, and could be abused if this script is used outside its intended context.

More critically, the script modifies system networking configuration based on an environment variable (`FARM`), which can be manipulated by an untrusted actor if the execution environment is not fully controlled. This introduces a supply chain risk: if an attacker can set `FARM=vmware`, they could force the use of a malicious DNS server in environments where 192.19.189.10 is not trusted or is unreachable, potentially leading to DNS spoofing or man-in-the-middle attacks during subsequent network operations.

Although the script performs standard early initialization tasks (mounting /proc, /sys, etc.), the unguarded write to `/etc/resolv.conf` using a non-standard, hardcoded DNS server constitutes a deviation from safe, reproducible system setup. This behavior could be leveraged maliciously if the script is repurposed or if the environment is compromised.
